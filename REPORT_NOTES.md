# Материалы для практического раздела НИР

3.1 и 3.2 уже написаны в отчете. Практическая реализация должна использовать их как вводную часть и далее показать, что разработанная методика действительно применима к выбранному веб-приложению, его зависимостям, Dockerfile, контейнерному образу и CI/CD-конвейеру.

Краткая связь практической части с теоретической частью: экспериментальный стенд подтверждает модель DevSecOps-конвейера за счет последовательной проверки выделенных артефактов. Исходный код анализируется средствами SAST, зависимости проверяются SCA-инструментом, секреты ищутся в коде и конфигурационных файлах, контейнерный образ проверяется Trivy, а результаты проверок используются в security gate для блокировки или разрешения сборки.

## 3.1 Назначение и состав экспериментального стенда

Что вставить в отчет:

Экспериментальный стенд реализован для практической апробации методики автоматизированной проверки безопасности веб-приложения в CI/CD-конвейере. Объектом проверки является минимальное веб-приложение на FastAPI. В состав стенда входят исходный код приложения, файл зависимостей `requirements.txt`, `Dockerfile`, конфигурация GitHub Actions, каталог отчетов и набор инструментов DevSecOps: Gitleaks, Bandit, Semgrep, pip-audit и Trivy.

Какая таблица нужна:

Таблица состава стенда: компонент, назначение, проверяемый артефакт, используемый инструмент.

Какие скриншоты сделать:

- структура проекта в файловом менеджере или IDE;
- открытый файл `README.md` со списком компонентов;
- каталог `.github/workflows` с файлом `security.yml`.

## 3.2 Архитектура экспериментального стенда

Что вставить в отчет:

Архитектура стенда соответствует модели, разработанной во втором разделе. После отправки изменений в GitHub запускается GitHub Actions workflow. Pipeline получает исходный код, выполняет поиск секретов, статический анализ, анализ зависимостей, сборку Docker-образа и сканирование образа. После выполнения проверок security gate анализирует результаты и принимает решение о блокировке или разрешении сборки.

Какую схему использовать:

Схема должна показывать последовательность: разработчик -> GitHub repository -> GitHub Actions -> Gitleaks/Bandit/Semgrep/pip-audit -> Docker build -> Trivy -> reports -> security gate -> pass/fail.

Какие элементы схемы пояснить:

- входные артефакты: `app/main.py`, `requirements.txt`, `Dockerfile`, `.github/workflows/security.yml`;
- классы проверок: secrets scanning, SAST, SCA, container scanning;
- выходные данные: JSON-отчеты и итоговое решение security gate.

## 3.3 Реализация тестового веб-приложения

Готовый текст для вставки в отчет:

Для проведения практической апробации было разработано тестовое веб-приложение на языке Python с использованием фреймворка FastAPI. Приложение выполняет роль объекта проверки в экспериментальном стенде и предназначено для демонстрации работы автоматизированных инструментов DevSecOps. При реализации приложения не ставилась задача создания полноценной бизнес-системы, поскольку основная цель практического раздела заключается в проверке разработанной методики на типовых артефактах веб-приложения.

Структура приложения включает основной файл `app/main.py`, файл моделей данных `app/models.py` и файл простого in-memory-хранилища `app/repository.py`. Такое разделение позволяет показать приложение как небольшой, но воспроизводимый backend-сервис: в отдельном модуле описаны Pydantic-модели входных и выходных данных, в отдельном модуле размещена логика хранения демонстрационных записей, а в основном модуле определены маршруты FastAPI. Использование базы данных в стенде не предусматривалось, так как оно не влияет на проверку методики и усложняет воспроизведение эксперимента.

В приложении реализованы служебные и демонстрационные endpoint. Endpoint `GET /` возвращает сведения о сервисе, `GET /health` используется для проверки работоспособности приложения, `GET /config` демонстрирует работу с конфигурационными параметрами, `GET /tasks` возвращает список демонстрационных задач, а `POST /tasks` позволяет добавить новую задачу в in-memory-хранилище. Дополнительно реализован endpoint `GET /security/artifacts`, который возвращает перечень проверяемых артефактов и соответствующих им классов проверок безопасности. Этот endpoint отражает связь практической реализации с теоретической частью, где были выделены исходный код, зависимости, конфигурационные файлы, Dockerfile и контейнерный образ.

Для первичной апробации методики в приложение намеренно добавлены тестовые дефекты безопасности. В файле `app/main.py` размещены fake credentials в формате, распознаваемом инструментом поиска секретов. Кроме того, endpoint `GET /diagnostics` содержит небезопасный вызов `subprocess.check_output` с параметром `shell=True`, что позволяет проверить работу инструментов SAST. Данные дефекты не предназначены для эксплуатации в реальной системе и используются только как контролируемые тестовые примеры для проверки security gate.

Описание FastAPI-приложения:

Веб-приложение реализовано на FastAPI и содержит небольшой набор endpoint, достаточный для проверки работоспособности стенда и демонстрации структуры backend API. Приложение не использует базу данных и внешние сервисы, так как цель практического раздела заключается не в разработке полноценного продукта, а в демонстрации автоматизированной проверки безопасности.

Какие endpoint реализованы:

- `GET /` - возвращает сведения о приложении;
- `GET /health` - используется для проверки доступности приложения;
- `GET /config` - демонстрирует чтение конфигурационных параметров;
- `GET /tasks` - возвращает список демонстрационных задач;
- `POST /tasks` - добавляет демонстрационную задачу в in-memory-хранилище;
- `GET /security/artifacts` - возвращает соответствие артефактов и инструментов проверки;
- `GET /diagnostics` - содержит тестовый дефект безопасности в первичной версии.

Какие файлы показать:

- `app/main.py`;
- `app/models.py`;
- `app/repository.py`;
- `requirements.txt`;
- `README.md`.

Какие скриншоты сделать:

- запущенное приложение в терминале через `uvicorn`;
- ответ endpoint `GET /health` в браузере или через `curl`;
- Swagger UI по адресу `/docs`;
- ответ endpoint `GET /security/artifacts`;
- фрагмент `app/main.py` с тестовыми дефектами до исправления.

## 3.4 Контейнеризация приложения

Готовый текст для вставки в отчет:

Для обеспечения воспроизводимого запуска приложения была выполнена контейнеризация с использованием Docker. Контейнеризация позволяет упаковать исходный код приложения, зависимости и параметры запуска в единый контейнерный образ. В контексте данной работы Dockerfile также рассматривается как самостоятельный проверяемый артефакт, поскольку ошибки в его настройках могут влиять на безопасность итоговой сборки.

Файл `Dockerfile` основан на базовом образе `python:3.11-slim-bullseye`. Данный образ выбран для первичного эксперимента намеренно: он позволяет продемонстрировать сканирование не только Python-зависимостей, но и системных пакетов внутри контейнерного образа. На первом этапе задаются переменные окружения Python и рабочий каталог `/app`. Затем в образ устанавливаются системные пакеты `build-essential`, `curl`, `git` и `openssh-client`. Наличие этих пакетов расширяет поверхность анализа и дает Trivy больше метаданных о компонентах образа. После этого в контейнер копируется файл `requirements.txt`, выполняется установка Python-зависимостей, затем копируется каталог `app`, содержащий исходный код FastAPI-приложения. Приложение запускается командой `uvicorn app.main:app --host 0.0.0.0 --port 8000`, что позволяет обращаться к API из хостовой системы через проброшенный порт.

Для уменьшения контекста сборки дополнительно используется файл `.dockerignore`. В него вынесены локальные служебные каталоги, отчеты сканирования, кэш Python и виртуальные окружения. Это делает сборку более предсказуемой и исключает попадание локальных артефактов анализа в контейнерный образ.

В первичной версии Dockerfile намеренно оставлены несколько особенностей, которые могут быть зафиксированы в ходе анализа контейнерного образа и его конфигурации: использование более старой ветки базового образа, установка дополнительных системных пакетов и запуск контейнера от пользователя `root`. Эти особенности используются как контролируемые тестовые условия для демонстрации того, что контейнеризация также должна входить в область автоматизированной проверки безопасности. В дальнейшем Dockerfile может быть улучшен путем сокращения набора системных пакетов, перехода на более свежий базовый образ и запуска приложения от непривилегированного пользователя.

Сборка образа выполняется командой `docker build -t devsecops-nir:demo .`. После сборки контейнер запускается командой `docker run --rm -p 8000:8000 devsecops-nir:demo`. Проверка доступности приложения выполняется запросом к endpoint `GET /health`. Таким образом, контейнеризация подтверждает, что тестовое приложение может быть воспроизведено в изолированной среде и затем проверено инструментом Trivy как контейнерный образ.

Описание Dockerfile:

Для контейнеризации используется `Dockerfile` на основе базового образа `python:3.11-slim-bullseye`. В образ устанавливаются дополнительные системные пакеты, копируется файл зависимостей, выполняется установка Python-пакетов, затем копируется каталог приложения. Контейнер запускает FastAPI-приложение через `uvicorn` на порту 8000. В первичной версии намеренно оставлены расширенная поверхность системных пакетов и запуск от пользователя `root`, чтобы проверить возможность фиксации проблем инструментами анализа контейнеров и конфигурации.

Команды сборки и запуска:

```bash
docker build -t devsecops-nir:demo .
docker run --rm -p 8000:8000 devsecops-nir:demo
curl http://127.0.0.1:8000/health
```

Какие скриншоты сделать:

- успешная сборка Docker-образа;
- запуск контейнера;
- ответ приложения из контейнера.
- содержимое `Dockerfile` и `.dockerignore`.

## 3.5 Реализация CI/CD-конвейера

Готовый текст для вставки в отчет:

CI/CD-конвейер для экспериментального стенда реализован с использованием GitHub Actions. Конфигурация pipeline размещена в файле `.github/workflows/security.yml`, который является частью репозитория и версионируется вместе с исходным кодом приложения. Workflow запускается при отправке изменений в ветку `main`, при создании pull request, а также вручную с помощью события `workflow_dispatch`. Такой способ запуска позволяет использовать pipeline как при обычной разработке, так и при проведении повторного эксперимента после исправления дефектов.

Конвейер состоит из одного задания `DevSecOps security gate`, выполняемого на виртуальной машине `ubuntu-latest`. На первом этапе выполняется получение исходного кода репозитория и настройка Python версии 3.11. Затем создается каталог `reports`, в который сохраняются результаты автоматизированных проверок. После подготовки окружения устанавливаются Python-инструменты Bandit, Semgrep и pip-audit.

Далее pipeline последовательно запускает инструменты, соответствующие классам проверок, выделенным в теоретической части работы. Gitleaks выполняет поиск секретов в исходном коде и конфигурационных файлах. Bandit выполняет статический анализ Python-кода и выявляет небезопасные конструкции. Semgrep используется как дополнительный SAST-инструмент, позволяющий проверить код по набору правил. pip-audit анализирует файл зависимостей `requirements.txt` и выявляет известные уязвимости в Python-пакетах. После этого выполняется сборка Docker-образа, который затем проверяется Trivy на наличие уязвимостей высокого и критического уровня. Отдельно Trivy запускается в режиме проверки конфигурации, что позволяет анализировать Dockerfile и другие инфраструктурные настройки.

Для реализации security gate каждый инструмент запускается с параметром `continue-on-error: true`. Это позволяет не останавливать pipeline на первом найденном дефекте, а выполнить все проверки и сформировать полный набор отчетов. Если конкретный инструмент завершился с ошибкой, его название записывается в файл `reports/security-gate.failures`. После выполнения всех проверок отчеты публикуются как artifact `security-reports`, а финальный шаг `Security gate` анализирует наличие записей в файле с ошибками. Если файл не пустой, сборка завершается с ошибкой, что соответствует решению `block` в разработанном алгоритме обработки результатов сканирования. Если блокирующие срабатывания отсутствуют, pipeline завершается успешно, что соответствует решению `allow`.

Таким образом, реализованный GitHub Actions workflow связывает результаты отдельных инструментов безопасности с формализованным решением о дальнейшем прохождении сборки. Это подтверждает практическую применимость разработанной методики: проверяемые артефакты анализируются соответствующими инструментами, результаты сохраняются в отчетах, а security gate выполняет автоматизированную блокировку сборки при обнаружении дефектов.

Описание GitHub Actions workflow:

CI/CD-конвейер реализован в файле `.github/workflows/security.yml`. Workflow запускается при push, pull request и вручную через `workflow_dispatch`. В ходе выполнения pipeline устанавливаются инструменты анализа, запускаются проверки Gitleaks, Bandit, Semgrep, pip-audit, выполняется сборка Docker-образа и сканирование Trivy. Результаты сохраняются в каталог `reports` и публикуются как artifact GitHub Actions.

Какие этапы pipeline реализованы:

- получение исходного кода из репозитория;
- установка Python и инструментов анализа;
- поиск секретов Gitleaks;
- SAST-анализ Bandit;
- дополнительный SAST-анализ Semgrep;
- SCA-анализ зависимостей pip-audit;
- сборка Docker-образа;
- сканирование Docker-образа Trivy;
- сканирование конфигурации Trivy;
- публикация отчетов;
- выполнение security gate.

Какие скриншоты сделать:

- список запусков workflow во вкладке GitHub Actions;
- детали первичного запуска с failed status;
- раскрытые шаги Gitleaks, Bandit, pip-audit и Trivy;
- artifact `security-reports`.

## 3.6 Проведение эксперимента с тестовыми дефектами безопасности

Какие дефекты были искусственно добавлены:

- fake secret в формате AWS-like credentials в `app/main.py`;
- небезопасный вызов `subprocess.check_output(..., shell=True)` в endpoint `/diagnostics`;
- уязвимая зависимость `urllib3==1.26.5` в `requirements.txt`;
- использование более старого базового образа `python:3.11-slim-bullseye` и дополнительных системных пакетов в `Dockerfile`;
- запуск контейнера от пользователя `root` в `Dockerfile`;
- контейнерный образ, собранный из Dockerfile и проверяемый Trivy.

Какой инструмент какой дефект обнаружил:

- Gitleaks должен обнаружить fake secret в формате AWS-like credentials;
- Bandit должен обнаружить использование `subprocess` с `shell=True`;
- Semgrep должен дополнительно зафиксировать небезопасную конструкцию в Python-коде;
- pip-audit должен обнаружить известные CVE в `urllib3==1.26.5`;
- Trivy должен проверить контейнерный образ на уязвимости системных пакетов и конфигурационные проблемы.

Какие скриншоты сделать:

- фрагмент кода с fake secret;
- фрагмент кода с `shell=True`;
- `requirements.txt` с уязвимой зависимостью;
- failed run в GitHub Actions;
- вывод каждого инструмента или соответствующий JSON-отчет из artifact.

| Исправление | Проверяемый артефакт | Инструмент | Ожидаемый результат |
| ----------- | -------------------- | ---------- | ------------------- |
| Удаление fake secret и использование переменной окружения | Исходный код, конфигурация | Gitleaks | Секрет не обнаружен |

Что было исправлено:

В повторной версии необходимо удалить hardcoded secret, заменить его чтением из переменной окружения `DEMO_API_TOKEN`, удалить endpoint `/diagnostics` или реализовать диагностику без `shell=True`, обновить зависимость `urllib3` до актуальной версии и изменить Dockerfile: сократить набор системных пакетов, при необходимости перейти на более свежий базовый образ и запускать приложение от непривилегированного пользователя.

Как результаты подтверждают методику:

Результаты первичного запуска показывают, что каждый класс проверок выявляет дефекты в соответствующем артефакте. Security gate блокирует сборку при обнаружении секретов и дефектов высокого уровня. Повторный запуск после исправлений демонстрирует, что pipeline может использоваться не только для обнаружения проблем, но и для подтверждения их устранения.

## Исправленная версия дефектных мест

Фрагмент `app/main.py` после исправления:

```python
import os

from fastapi import FastAPI

app = FastAPI(title="DevSecOps NIR Demo")


@app.get("/")
def read_root() -> dict[str, str]:
    return {"message": "DevSecOps NIR demo application"}


@app.get("/health")
def health() -> dict[str, str]:
    return {"status": "ok"}


@app.get("/config")
def read_config() -> dict[str, str]:
    token = os.getenv("DEMO_API_TOKEN", "")
    return {
        "mode": os.getenv("APP_MODE", "demo"),
        "token_source": "environment",
        "token_configured": str(bool(token)).lower(),
    }
```

Фрагмент `requirements.txt` после исправления:

```text
fastapi==0.115.12
uvicorn[standard]==0.34.2
urllib3==2.5.0
```

Фрагмент `Dockerfile` после исправления:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt \
    && addgroup --system appgroup \
    && adduser --system --ingroup appgroup appuser

COPY app ./app

USER appuser

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```
